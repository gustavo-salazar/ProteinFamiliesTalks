## Accessing InterPro programmatically _Gustavo A. Salazar_
## A REST API https://www.ebi.ac.uk/interpro/api/ ```txt [ Client ] || /\ URL || || JSON || || \/ || [ Server ] ```
## The entry endpoint The root of the endpoint tells us the number of entries per member database https://www.ebi.ac.uk/interpro/api/entry
## The entry endpoint https://www.ebi.ac.uk/interpro/api/entry/interpro Including `/interpro` as our database, has 2 effects: * Filters the result to only include the entries with database `interpro` * Displays the result as a list of entities
## The entry endpoint https://www.ebi.ac.uk/interpro/api/entry/interpro/IPR000001 Including `/IPR000001` as our accession, has 2 effects again: * Selects that entity in particular * Displays the result as a JSON object with the metadata of that entity
## URL structure All the endpoints follow the same structure `/api/`**`[endpoint]/`** It returns an aggregatted view of the endpoint, showing how many items per database. --- `/api/`**`[endpoint]/`** **`[database]/`** Generates the list of items in the endpoint for the seleccted DB --- `/api/`**`[endpoint]/`** **`[database]/`** **`[accession]/`** It gives the metadata of the entity with that acceession.
## Long Lists When using a URL that returns a list of items. It is likely that the number of results is higher than 20, which is the default size of InterPro API responses. You can change this value using the parameter `page_size` to a maximum of 200. E.g. https://www.ebi.ac.uk/interpro/api/entry/sfld?page_size=200
## Long Lists If the list is bigger than 200 items, then multiple HTTP requests are required to get the full set. You can get the link to the next page, from the current page in its key `next`. e.g. [/api/entry/sfld?page_size=200](https://www.ebi.ac.uk/interpro/api/entry/sfld?page_size=200) ``` { "next": "https://www.ebi.ac.uk/interpro/api/entry/sfld?cursor=cD1TRkxERzAxMDk1&page_size=200", ... } ```

## Mixing endpoints Interpro API allows you to use multiple endpoints. Firstly, we will call _[ENDPOINT BLOCK]_ to the group of URL levels that relate to a particular endoint: **`[endpoint]/`** **`[database]/`** **`[accession]/`** So, it is possible to put together more than one _[ENDPOINT BLOCK]_. * `/api/`_[ENDPOINT BLOCK]_ `/` _[ENDPOINT BLOCK]_ `/`
## Mixing endpoints The first _[ENDPOINT BLOCK]_ defines the main endpoint and output of the response. Any other _[ENDPOINT BLOCK]_ will be considered as a filter. e.g. [/api/protein/uniprot/entry/interpro/IPR000001](https://www.ebi.ac.uk/interpro/api/protein/uniprot/entry/interpro/IPR000001)
## Mixing endpoints [/api/protein/uniprot/entry/interpro/IPR000001](https://www.ebi.ac.uk/interpro/api/protein/uniprot/entry/interpro/IPR000001) * First _[ENDPOINT BLOCK]_: `/protein/uniprot` * Second _[ENDPOINT BLOCK]_: `/entry/interpro/IPR000001` --- Returns the list of uniprot proteins that are related (i.e. Have a match) with the interpro entry with accession `IPR000001`.

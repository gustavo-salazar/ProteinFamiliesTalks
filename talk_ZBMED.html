InterPro API

#### A REST API __API__: Application programming interface __REST__: Representational State Transfer ```txt [ Client ] || /\ URL || ||JSON || || \/ || [ Server ] ```
https://www.ebi.ac.uk/interpro/api/ https://www.ebi.ac.uk/interpro/wwwapi/ --- ##### Endpoints 1. `/entry` 2. `/protein` 3. `/structure` 4. `/taxonomy` 5. `/proteome` 6. `/set`
#### URL structure All the endpoints follow the same structure. --- `/api/`**`[endpoint]/`** **`[database]/`** **`[accession]/`** --- ##### Example [/api/entry/prosite/PS00673](https://www.ebi.ac.uk/interpro/api/entry/prosite/PS00673)

Mixing endpoints

/api/ [ENDPOINT BLOCK] / [ENDPOINT BLOCK] /

  • The first [ENDPOINT BLOCK] defines the main endpoint and output of the response.
  • Any other [ENDPOINT BLOCK] will be considered as a filter.

Mixing endpoints

e.g. /api/protein/uniprot/entry/interpro/IPR000126

Returns the list of uniprot proteins that are related (i.e. Have a match) with the interpro entry with accession IPR000126.

Release Procedure
  • We use Redis.
  • The procedure takes around 1-2 days
  • 3 phases:
    1. Get the URLs used from previous release
    2. Runs those URLs on the staging API machine (wp-np3-ac)
    3. Copies the cache file into the live Redis instance
  • Once released, every new API query is saved in our cache.
